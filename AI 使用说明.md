# API 通知系统设计 —— AI 使用说明

本文说明在本方案设计过程中，AI 的参与方式、被采纳与未采纳的建议，以及关键设计决策的来源。

---

## 一、AI 在哪些关键地方提供了帮助

### 1. 辅助需求拆解

通过多轮需求沟通，AI 帮助我把题目从“调用外部 HTTP API”拆解为几个工程问题：统一入口、第三方协议隔离、可靠投递、失败重试、幂等控制和后续演进。

---

### 2. 架构图生成

AI 基于需求生成了初版架构图和时序图。我根据 MVP 的实现范围进行删减和调整，最终保留了接入层、服务层、本地消息表、调度器、防腐层和外部系统这条主链路。

---

### 3. 演进路径设计

AI 提供的演进框架：
- DB Queue → MQ → 事件驱动
- 单机调度 → 分片调度 → 多租户治理

### 4. 基础代码实现

AI 根据设计文档，按功能点快速生成了基础代码结构和单元测试。我主要检查了可靠性语义、边界是否清晰，以及实现是否符合 MVP 的复杂度。

---

## 二、AI 提供但未被采纳的建议

### 1. 引入消息队列（Kafka）

AI 建议：

在设计初期直接引入 MQ 解耦。

未采纳原因：

- MVP 阶段优先验证“请求先落库、异步投递、失败可重试”的可靠投递闭环，本地消息表已经足够。
- 直接引入 MQ 会增加部署、运维、监控和消费一致性的成本。
- 如果未来确实引入 MQ，我会更倾向 RocketMQ 而不是 Kafka。通知消息体通常不大，且后续如果需要优先级队列、延迟消息等能力，RocketMQ 的业务适配成本更低；Kafka 更适合高吞吐日志流场景，优先级和延迟语义往往需要更多业务代码补齐。

### 2. 复杂调度系统（优先级 + 动态流控）

AI 建议：

- priority queue
- 多级调度策略
- 动态 QPS 控制

未采纳原因：

- 当前业务未出现明显差异化优先级需求
- 属于“规模驱动型设计”，不是“问题驱动型设计”
- 当前代码仅保留 `priority` 字段和查询排序，作为未来扩展点

### 3. 完整可观测 + 告警体系

AI 建议：

- 全链路 tracing
- vendor 维度告警体系

未采纳原因：

- MVP 阶段价值不匹配成本
- 优先验证“能否可靠送达”
- 当前先保留结构化日志和死信状态，后续再补 Prometheus 指标、告警和链路追踪

## 三、我做出的关键决策

### 1. 投递语义选择 At Least Once，而不是 Exactly Once

这是我自己做出的核心可靠性判断。

外部 HTTP 调用存在超时、连接中断等不确定状态：调用方无法准确判断第三方系统是否已经处理成功。如果强行承诺 Exactly Once，系统会引入很高复杂度，也仍然无法绕开第三方接口幂等的问题。

因此我选择 At Least Once：通知服务保证消息不轻易丢失，并在失败时重试；第三方 API 需要通过业务幂等键处理重复请求。这更符合真实 HTTP 集成场景。

### 2. MVP 使用本地消息表，而不是一开始引入 MQ

AI 曾建议直接使用 Kafka/MQ，但我选择第一版使用数据库本地消息表。

原因是本题更关注工程判断和取舍，而不是功能堆叠。对 MVP 来说，请求先写入 `notification_task` 表，再由调度器异步投递，已经可以覆盖“服务重启不丢消息、失败可重试、最终进死信”的核心目标。

MQ 更适合在流量显著增长、削峰填谷诉求明显、多消费者并发投递成为瓶颈时引入。第一版直接上 MQ 会让系统边界、部署和运维复杂度过早上升。

### 3. 使用防腐层隔离第三方协议

不同供应商的 URL、Header 和 Body 格式不同。如果让业务系统直接理解这些差异，后续会形成强耦合。

因此我保留统一领域模型 `NotificationMessage`，再通过 `Converter` 将其转换为供应商请求；`ConverterFactory` 根据 `TargetSystem` 选择对应转换器。这个决策可以让新增供应商时主要新增 Converter，而不是改动核心投递流程。

### 4. 使用 requestId 做接入幂等

通知服务采用 At Least Once 后，重复提交和重复投递都是需要接受并管理的现实。

我选择让业务方提交 `requestId`，并在数据库层建立唯一约束。这样重复请求可以直接返回已有任务，避免同一个业务事件在通知系统内被创建多条任务。

这里没有设计复杂的幂等表或分布式锁，是因为 `requestId` 唯一索引已经能覆盖 MVP 的核心问题，而且失败时数据库约束也是最后一道保护。

### 5. 设置有限重试和死信，而不是无限重试

失败处理上，我选择 `1 / 5 / 30 / 60 / 360` 分钟的退避节奏，并设置最大重试次数。

这样做的原因是外部系统短暂抖动时可以自动恢复；如果对方长期不可用，无限重试只会持续占用系统资源，甚至放大故障。进入 `DEAD` 状态后转人工处理，边界更清晰，也便于后续接入告警和补偿工具。

### 6. 明确系统边界：只负责可靠送达，不负责第三方业务语义

我没有让通知系统处理 CRM 状态是否合法、库存是否足够、广告系统是否认可这类业务语义。

通知系统的职责是接收业务系统提交的通知请求，并尽可能可靠地投递到目标地址。第三方返回业务失败时，在 MVP 中统一按投递失败记录和重试/死信处理；更细的业务补偿应由对应业务域或人工流程处理。

### 7. 对外接入选择 HTTP API

第一版对业务系统暴露 `POST /api/v1/notifications`，而不是强依赖某个语言 SDK。

HTTP 的优势是语言无关、接入成本低，也方便统一做认证、审计、限流和监控。未来如果内部接入方增多，可以再提供 Java/Go SDK，但 SDK 应只是便利层，底层仍调用统一 HTTP API。

### 8. 接受 MVP 的限制，并说明未来演进方向

当前实现主要面向单实例或低并发场景：调度器通过数据库轮询拉取任务，暂未实现多实例抢占锁、供应商维度隔离、动态流控和完整可观测。

这些不是忽略，而是我有意放在后续演进阶段：当流量增长或供应商稳定性差异变明显时，再引入 MQ、分片调度、供应商级限流、Prometheus 指标、告警和死信回放。这样第一版可以保持简单，同时保留清晰的扩展路径。
